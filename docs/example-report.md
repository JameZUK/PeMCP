## 1. Executive Summary

This report documents the analysis of a **BYOVD (Bring Your Own Vulnerable Driver) loader/dropper** that drops a weaponised AMI BIOS flash utility and multiple kernel drivers to achieve kernel-level access on victim machines. The malware masquerades its payloads as NVIDIA and AMD GPU services in trusted Windows system directories. It deletes volume shadow copies and stops the WMI service before executing its payload, consistent with pre-ransomware or destructive activity preparation.

The loader embeds **5 plaintext PE files** (no encryption) in its `.data` section, totalling 939KB. The primary payload is a modified AMI DMI Edit utility that communicates via a named pipe (`\\.\Pipe\UC2MessagePool`) and loads vulnerable kernel drivers to obtain arbitrary physical memory read/write access via `MmMapIoSpace`.

Portuguese-language strings in the binary suggest **Brazilian origin**.

## 3. Execution Flow

### 3.1 Stage 1 — Loader (this binary)

The entry point reaches **sub_14000c700** (orchestration entry), which:

1. **Sets console window 90% opaque** (alpha 230) to reduce visual footprint
2. **Decodes XOR stackstrings** using key `(char)(i + 121)` — short strings like `"cls"` hidden at build time
3. **Calls the main orchestrator** at **sub_1400090e0** (824 basic blocks, the largest function)
4. **Deletes all files in `C:\Windows\prefetch\`** using `FindFirstFileW`/`DeleteFileW` to destroy forensic evidence
5. Prints Portuguese status messages: *"Arquivo excluído"* (file deleted) / *"Falha ao excluir arquivo"* (failed to delete)

### 3.2 Main Orchestrator (sub_1400090e0)

The orchestrator executes the following sequence:

#### Anti-Forensics Commands
Executed via `sub_140018ce0` (CreateProcessW wrapper with UTF-8 codepage 65001):

| Command | Purpose |
|---------|---------|
| `vssadmin delete shadows /All /Quiet >NUL 2>&1` | Delete all volume shadow copies |
| `net stop winmgmt /Y >NUL 2>&1` | Stop WMI service (disable logging/detection) |
| `cls` (XOR-decoded) | Clear console output |

#### Payload Drops
All payloads written via C++ iostream `stream.write()` (sub_14000c120) — raw unencrypted PE data from `.data` section:

**Set A — NVIDIA masquerade** (`C:\Windows\IME\`):

| File | Size | .data VA |
|------|------|----------|
| NVContainer.exe | 386KB (0x5e670) | 0x1400cf770 |
| amifldrv64.sys | 29KB (0x7180) | ~0x1400c85ea |
| amigendrv64.sys | 35KB (0x8788) | ~0x1400bfe5a |

**Set B — AMD masquerade** (`C:\Windows\Performance\`):

| File | Size | Notes |
|------|------|-------|
| AMDContainer.exe | 386KB | Same binary as NVContainer.exe |
| amifldrv64.sys | 29KB | Same driver, second location |

#### Payload Execution
Launches NVContainer.exe with command flags built via `sub_140016e90`:

```
C:\Windows\IME\NVContainer.exe /SU auto /PSN <ID> /SS <ID> /SV <ID> /BM <ID> /PAT <ID> /PPN <ID>
```

Each `<ID>` is a **15-character random string** generated by MT19937-64 PRNG (`sub_140007da0`) using the charset `"0123456789ACBDEF"` (note: B and C transposed vs standard hex). The PRNG is seeded from `GetTickCount`.

### 3.4 Stage 3 — Kernel Drivers (BYOVD)

Three variants of the legitimate AMI BIOS flash driver are embedded for broad hardware/OS compatibility:

| File | SHA-256 | Size | Arch | Imports |

| amigendrv64.sys | `38d87b51...` | 35KB | 64-bit | 30 |

All three are:
- **Digitally signed** with DigiCert EV Code Signing certificates (legitimate but abused)
- **Native/Kernel-mode** (Subsystem: Native)
- Import `ntoskrnl.exe` + `HAL.dll`

**Critical kernel APIs:**

| API | Capability |
|-----|------------|
| `MmMapIoSpace` | Maps physical memory into kernel virtual space — **arbitrary physical memory R/W** |
| `ZwMapViewOfSection` | Memory section mapping (process injection primitive) |
| `IoCreateDevice` | Creates a device object accessible from usermode |
| `IoCreateSymbolicLink` | Creates symbolic link for usermode access via `\\.\Global\%s` |
| `ObReferenceObjectByHandle` | Object handle manipulation |

**Attack primitive:** The driver provides `MmMapIoSpace`, which allows NVContainer.exe to read and write arbitrary physical memory from usermode. This can be used to patch kernel structures, disable security callbacks, or manipulate BIOS/firmware.

## 8. Analysis Corrections

During deep analysis, two initial automated assessments were corrected:

### RC4 — False Positive
Initial triage identified 9 RC4 identity permutation instances and concluded the loader uses RC4 to encrypt embedded payloads. **This is incorrect.** All 9 instances exist within the embedded payload PEs themselves (NVContainer.exe's crypto code and driver certificate data). The loader performs **no cryptographic decryption**. Payloads are stored as raw, unencrypted PE files with visible MZ headers.

### Capa XOR — False Positive
Capa flagged 5 addresses as "encode data using XOR." All 5 are **Mersenne Twister MT19937 PRNG operations**:
- `0x140004b00`: MT init loop (constant `0x6c078965`, 624 iterations)
- `0x140004f90`, `0x140004ff0`, `0x140005040`: MT state manipulation
- `0x140007e90`: MT twist operation

The only actual encoding in the loader is a trivial XOR stackstring technique (key: `(char)(i + 121)`, 3 bytes) to hide the string `"cls"`.

## 9. Detection Recommendations

### YARA
- Rich header hash: `24de37666127cde6de214e61914391a6`
- String: `"\\.\Pipe\UC2MessagePool"` (unique named pipe)
- String: `"0123456789ACBDEF"` (B/C-transposed hex charset — distinctive)
- Byte sequence at orchestrator: hardcoded paths `"C:\Windows\IME\NVContainer.exe"`, `"C:\Windows\Performance\AMDContainer.exe"`

### Network/Host
- Monitor for named pipe creation: `\\.\Pipe\UC2MessagePool`
- Monitor for AMI driver service installation (`amifldrv64`, `amigendrv64`)
- Alert on files dropped to `C:\Windows\IME\` or `C:\Windows\Performance\` with PE headers
- Alert on `net stop winmgmt` followed by `vssadmin delete shadows`

### Driver Blocklist
- Block loading of `amifldrv64.sys` (SHA-256: `20f11a64...`) and `amigendrv64.sys` (SHA-256: `38d87b51...`) via Microsoft's Vulnerable Driver Blocklist or WDAC policy

## 10. Conclusion

This is a **multi-stage BYOVD loader** of moderate sophistication. The attack chain is straightforward: drop payloads, install a signed vulnerable driver as a service, and use it for kernel-level access. The malware's operational security is mixed — it takes anti-forensic steps (shadow copy deletion, WMI shutdown, prefetch cleanup) but stores its payloads as plaintext unencrypted PEs, making static extraction trivial.

The use of a modified AMI DMI Edit utility as the usermode component, combined with the `UC2MessagePool` named pipe and service installation flags (`/SU auto`, `/PSN`, `/SS`, `/SV`), suggests this is part of a larger framework. The dual-vendor masquerade (NVIDIA + AMD) and dual drop locations indicate an operator concerned with persistence and redundancy.

The Portuguese-language strings and the VT label "trojan.delshad" place this in the context of **Brazilian cybercrime operations** that commonly employ BYOVD techniques for banking trojans and ransomware preparation.
