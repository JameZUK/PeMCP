# Analysis Report: Ternary Trap — Neural Network Crackme (Deep RE Showcase)

**Prompt:** `/arkana-analyse Find a new tricky crackme and go ahead and crack it`

**Sample:** PolyMLP.exe by Leenear (PE32+ x64, 54,272 bytes) + model.nli (88,252 bytes) — [crackmes.one](https://crackmes.one/crackme/69a54bd70f5b9757a6a5f72f) | Difficulty 4.0/6.0

**Password:** `3ig3nv4lu3` (leet-speak: "eigenvalue")
**Flag:** `NLI{w31ghts_4r3_th3_k3y}` (leet-speak: "weights are the key")

---

## Why This Example

This analysis demonstrates Arkana's **deep reverse engineering** capabilities on
a novel challenge that sits at the intersection of machine learning and binary
analysis. The crackme encodes its password not as a string, hash, or
mathematical formula, but as the deterministic output of a **ternary neural
network** stored in a custom encrypted binary format. Solving it required:

- Reversing a custom file format (NLI2) with XOR encryption and integrity
  hashing
- Understanding a 2-layer MLP (multi-layer perceptron) with ternary weights
  {-1, 0, +1}
- Tracing SIMD assembly (`cvtdq2pd`, `mulsd`, `addsd`, `addpd`) to determine
  the exact numerical computation
- Implementing autoregressive character-level inference to reproduce the
  model's predictions
- Using depth-first search guided by the model to find the password

No emulation or debugging was needed — pure static analysis and mathematical
reconstruction. The entire analysis and solve was completed in a single session
using 50+ Arkana tool invocations.

PolyMLP.exe implements a "Neural Password Lock" that loads a companion
`model.nli` file containing a trained neural network. The password is not stored
anywhere in the binary or data file — it exists only as the deterministic
autoregressive output of the network. The program reads a 10-character password,
evaluates its probability under the model, then generates 5 validation
characters. If the generated output spells "VALID" (byte sum = 368), access is
granted and a 24-character flag is generated via a second inference pass.

## Phase 3 — Deep Dive: Reversing the Main Loop

### 3.1 Programme Flow (0x1400038d0)

`decompile_function_with_angr` at 449 lines across 4 pages revealed the
complete validation logic:

1. **Load model**: `sub_1400024a0(&model, "model.nli")` — returns struct with
   weights, biases, and parameters
2. **Display parameters**: V, K, H, and total parameter count
3. **Compute XOR checksums**: `pw_checksum = field_8c ^ 0xC0FFEE`,
   `flag_checksum = field_90 ^ 0xDECADE`
4. **3-attempt password loop**:
   - Read password from stdin, pad to `pw_len` characters
   - `sub_140003150` — evaluate password probability against model
   - Append START token (`\x03`), run `sub_140003570` to generate 5 characters
   - SIMD-accelerated byte summation of generated output
   - **Two checks**: float threshold comparison AND byte sum = `pw_checksum`
5. **Flag generation**: combine password + START + validation output + DELIM,
   generate 24 characters, verify against `flag_checksum`

The XOR constants `0xC0FFEE` and `0xDECADE` are classic CTF signatures —
"coffee" and "decade".

### 3.2 Model Loader (0x1400024a0)

The 647-line model loader function revealed the NLI2 binary format:

**XOR Decryption**: The entire file is decrypted with a 4-byte key cycling
every byte (`data[i] ^= key[i & 3]`). The key is stored at `g_14000a8f0` in
the `.rdata` section.

**Integrity Check**: A rolling hash initialised with `0xDEADBEEF`:
```
hash = 0xDEADBEEF
for each byte b (except last 4):
    hash = b ^ (hash >> 27) ^ (hash * 32)
assert hash == last_4_bytes_of_file
```

**Magic Validation**: Byte-by-byte comparison against `NLI2` (0x4E, 0x4C,
0x49, 0x32) — confirmed via assembly at `0x14000286e`–`0x140002888`.

**Weight Loading**: The critical discovery was in the assembly at
`0x140002970`:
```asm
movsx  eax, byte ptr [rdi]       ; sign-extend byte (-1, 0, or +1)
movd   xmm0, eax                 ; move to SSE register
cvtdq2pd xmm0, xmm0              ; int32 → float64 conversion
movsd  qword ptr [...], xmm0     ; store as double-precision
```

The `cvtdq2pd` instruction was the key insight — the decompiler showed
`*((long long*)dst) = *((char*)src)` which hid the integer-to-float conversion.
Without verifying against assembly, the numerical computation would have been
entirely wrong.

### 3.3 Determining the XOR Key

With the model loaded as raw binary (`open_file(mode='shellcode')`), the first
4 bytes were read via `get_hex_dump`: `90 E1 F7 DD`.

Since the decrypted magic must be "NLI2" (`4E 4C 49 32`):
```
key[0] = 0x90 ⊕ 0x4E = 0xDE
key[1] = 0xE1 ⊕ 0x4C = 0xAD
key[2] = 0xF7 ⊕ 0x49 = 0xBE
key[3] = 0xDD ⊕ 0x32 = 0xEF
```

**XOR key = `0xDEADBEEF`** — the same constant used for the integrity hash
initialisation. An elegant design choice by the author.

## Phase 4 — Deep Dive: The Neural Network

### 4.1 Input Encoder (0x140002310)

The input encoder converts K=5 context characters into a V×K = 140-dimensional
**one-hot** vector:

```
For each position i ∈ [0, K):
    Look up character in binary search tree → index j
    Set input[V×i + j] = 1.0 (IEEE 754: 0x3FF0000000000000)
    All other positions remain 0.0
```

Characters not found in the charmap (e.g. null padding) produce zero vectors —
effectively treating unknown characters as "no input" for that position.

### 4.2 MLP Forward Pass (0x140001880)

The 500-line forward pass implements a standard 2-layer MLP with SIMD
acceleration:

```
Input:   x ∈ ℝ¹⁴⁰  (one-hot encoded context)
Layer 1: h = ReLU(x·W₁ + b₁)    where W₁ ∈ {-1,0,+1}¹⁴⁰ˣ⁵¹² , b₁ ∈ ℝ⁵¹²
Layer 2: logits = h·W₂ + b₂     where W₂ ∈ {-1,0,+1}⁵¹²ˣ²⁸  , b₂ ∈ ℝ²⁸
Output:  argmax(logits) → predicted character index
```

Assembly verification (`get_annotated_disassembly`) confirmed:
- **`mulsd`** / **`addsd`** — scalar double-precision multiply-accumulate
- **`addpd`** — packed double-precision for vectorised bias addition
- **`MaxV`** — SIMD max operation implementing ReLU: `max(0, h_i)`

With ternary weights, the matrix multiplication reduces to additions and
subtractions — each input element (0 or 1) either adds, subtracts, or skips the
corresponding weight.

### 4.3 Autoregressive Inference (0x140003570)

The inference function generates characters one at a time using a sliding
context window:

```
context ← last K characters of input
for step = 1 to num_chars:
    x ← one_hot_encode(context)
    logits ← forward_pass(x)
    predicted_char ← charmap[argmax(logits)]
    append predicted_char to output
    context ← context[1:] + [predicted_char]   (slide window)
```

This is a standard **character-level language model** inference loop. The model
is deterministic — given the same context, it always produces the same output.

### 4.4 Bias Type Discovery

An early solver attempt produced all-zero outputs because biases were parsed as
`int32` instead of `float32`. The raw bias values `-1082130432` and
`1065353216` were the giveaway:

```
-1082130432 = 0xBF800000 → IEEE 754 float32 = -1.0
 1065353216 = 0x3F800000 → IEEE 754 float32 = +1.0
```

The decompiler showed `*((int*)cur)` for the bias load, masking the fact that
the assembly uses the same `cvtdq2pd` conversion pattern as the weights. The
biases, like the weights, are ternary: {-1.0, 0.0, +1.0}.

## Phase 5 — Solving the Password

### 5.1 Key Insight: Context Window Determines Everything

Since the model uses a sliding context window of K=5 characters, and the input
to the inference function is the 10-character password + START token (11
characters total), only the **last 5 characters** of the input form the initial
context. This means:

- Characters at positions 6–9 of the password + the START token determine the
  generated validation output
- Characters at positions 0–5 affect only the log-probability threshold check

### 5.2 The Log-Probability Constraint

`sub_140003150` evaluates the password by running the model at each position
and computing the log-probability that the model would have predicted the actual
character. The **average log-probability must exceed -0.25** — meaning each
character must be the model's top prediction with high confidence.

This constraint means the password must be the model's own **natural
autoregressive output** starting from a zero-padded context (simulating the
evaluation from the beginning of the string where preceding characters are
unknown).

### 5.3 DFS Solution

Following the challenge hint ("Use DPS — depth-first search"), the solver
builds the password character by character, always following the model's
argmax prediction:

```
Position 0: context = [-1,-1,-1,-1,-1] (all padding) → model predicts '3'
Position 1: context = [-1,-1,-1,-1, 0] → model predicts 'i'
Position 2: context = [-1,-1,-1, 0, 1] → model predicts 'g'
Position 3: context = [-1,-1, 0, 1, 2] → model predicts '3'
Position 4: context = [-1, 0, 1, 2, 0] → model predicts 'n'
Position 5: context = [ 0, 1, 2, 0, 3] → model predicts 'v'
Position 6: context = [ 1, 2, 0, 3, 4] → model predicts '4'
Position 7: context = [ 2, 0, 3, 4, 5] → model predicts 'l'
Position 8: context = [ 0, 3, 4, 5, 6] → model predicts 'u'
Position 9: context = [ 3, 4, 5, 6, 7] → model predicts '3'
```

**Password: `3ig3nv4lu3`** — leet-speak for **"eigenvalue"**, a beautiful
mathematical reference. In linear algebra, eigenvalues characterise the
fundamental behaviour of matrices — precisely what defines this neural
network's weight matrices.

### 5.4 Validation

With the password determined, the full validation chain was verified:

```
Input to inference: "3ig3nv4lu3" + START(0x03)
Context window: [u(7), 3(0), START(8)] padded to K=5
Generated 5 characters: V, A, L, I, D → "VALID"
Byte sum: 86 + 65 + 76 + 73 + 68 = 368  ✓ (matches 0xC0FE9E ⊕ 0xC0FFEE)
Log-probability: 0.0000 (perfect — every character predicted with 100% confidence)
Threshold check: 0.0 > -0.25  ✓
```

| **Checksum XOR masks** | `0xC0FFEE` (password) and `0xDECADE` (flag) |

## Key Techniques Demonstrated

### 1. Assembly Cross-Checking for Type Discovery

The decompiler represented weight loading as `*((long long*)dst) = *((char*)src)`,
which would store a sign-extended integer into 8 bytes. Reading this as float64
would yield denormalised values (≈ 0) or NaN — making the neural network
non-functional.

The assembly revealed the hidden `cvtdq2pd` instruction that converts the
integer to a proper float64. **This is precisely the scenario described in
Arkana's mandatory assembly cross-check rules** — the decompiler lost critical
type information that only the raw instructions preserved.

Similarly, biases appeared as int32 values (`-1082130432`, `1065353216`) in the
decompiler output. Recognising these as IEEE 754 float32 encodings of -1.0 and
+1.0 was essential.

### 2. Multi-File Analysis via File Switching

The analysis required switching between the PE binary and the model data file:
- **PE loaded**: decompilation, disassembly, string search, import analysis
- **model.nli loaded as raw**: hex dump, XOR decryption with `refinery_xor`
- **PE reloaded**: continued decompilation from cached session

Arkana's per-file disk cache preserved all analysis state across file switches,
allowing seamless continuation.

### 3. Custom Binary Format Parsing

The NLI2 format was fully reversed using a combination of:
- `decompile_function_with_angr` — structural understanding
- `get_annotated_disassembly` — type verification (cvtdq2pd discovery)
- `refinery_xor` — in-tool decryption of specific file regions
- `get_hex_dump` — raw byte inspection at computed offsets

### 4. Section-Aware Hex Dump

Locating the XOR key required computing the file offset from the virtual
address. The key was referenced as `g_14000a8f0` (RVA `0xA8F0`), which maps
to file offset `0xA8F0 - 0xA000 + 0x8A00 = 0x98F0` based on the `.rdata`
section's virtual address (`0xA000`) and raw offset (`0x8A00`). `get_pe_data`
provided the section table for this computation.

## Solver Code

The final solver was implemented as a standalone Python script (120 lines) that:

1. Reads `model.nli` and XOR-decrypts with `0xDEADBEEF`
2. Parses the NLI2 header, ternary weights, float32 biases, and character map
3. Implements the 2-layer MLP forward pass with ReLU activation
4. Runs autoregressive DFS from zero-padded context
5. Verifies both the password checksum and flag checksum

```python
# Core forward pass (8 lines of numpy)
def forward(model, context_indices):
    x = np.zeros(V * K)
    for i, idx in enumerate(context_indices):
        if 0 <= idx < V:
            x[V * i + idx] = 1.0
    h = np.maximum(x @ W1 + bias1, 0.0)       # ReLU hidden layer
    logits = h @ W2 + bias2                     # output logits
    return int(np.argmax(logits)), logits       # argmax → character

# DFS password search (follows model's top predictions)
password = []
for pos in range(10):
    ctx = password[max(0, pos-K):pos]
    ctx = [-1] * (K - len(ctx)) + ctx           # zero-pad
    pred_idx, _ = forward(model, ctx)
    password.append(pred_idx)

# Result: [0,1,2,0,3,4,5,6,7,0] → "3ig3nv4lu3"
```

The solver confirms: average log-probability = 0.0000 (every character predicted
with 100% confidence), validation output = "VALID" (sum 368), flag =
`NLI{w31ghts_4r3_th3_k3y}` (sum 2184).
